1. Sitio estático en Cloudflare Pages

Completado HTML / CSS / JS Sin build step

Archivosindex.html, styles.css, script.js, README.md

Pasos realizados

  1. Crear archivos HTML/CSS/JS en el directorio local.
  2. Inicializar git y hacer primer commit en rama main.
  3. Crear repositorio público en GitHub y pushear el código.
  4. En el dashboard de Cloudflare: Workers & Pages → Create → Pages → Connect to Git.
  5. Autorizar la app de Cloudflare en GitHub y seleccionar el repo.
  6. Configurar el proyecto sin framework, sin build command, output dir /.
  7. Verificar deploy automático haciendo un cambio + git push.

Diagrama del flujo

GitHub repo (main) ──push──▶ Cloudflare Pages ──serve──▶ *.workers.dev tu código build automático + deploy red global edge

Aprendizajes clave

  • Pages es hosting estático en la red global de Cloudflare con CI/CD gratis.
  • Cada push a la rama de producción dispara un nuevo deploy.
  • Pushes a otras ramas generan preview deploys con URL única (útil para revisar antes de mergear).
  • El SSL/HTTPS, CDN global y headers básicos vienen incluidos sin configurar nada.
  • Si el sitio es HTML puro: sin build command y output dir /. Si usa Astro/Next/etc, se configura el build.
  • Cloudflare unificó Pages con Workers (Static Assets): por eso el dominio resultó *.workers.dev en vez del histórico *.pages.dev. Funcionalmente equivalente.

Setup que se realizó

# 1. Crear estructura
mkdir AcortadorURL && cd AcortadorURL
npm init -y

# 2. Instalar y loguear Wrangler
npm install -D wrangler@4
npx wrangler login                    # abre navegador para autorizar

# 3. Crear KV namespace
npx wrangler kv namespace create LINKS
# → guardar el id en wrangler.toml

# 4. Setear secret de producción
npx wrangler secret put AUTH_TOKEN    # pide ingresar valor (no se ve)

# 5. Crear .dev.vars (para dev local, NUNCA commitearlo)
echo 'AUTH_TOKEN="..."' > .dev.vars

# 6. Probar local
npm run dev                            # http://localhost:8787

# 7. Deploy a producción
npm run deploy

Aprendizajes clave

  • Workers = código JS ejecutándose en el edge global, no solo servir archivos.
  • KV = almacenamiento clave-valor distribuido, consultado con env.LINKS.get/put.
  • Bindings: wrangler.toml conecta recursos (KV, vars, secrets) al código vía env.X. Sin red, sin SDK — el binding es la interfaz.
  • Wrangler dev simula todo el entorno (incluyendo KV) en local sin tocar producción ni gastar requests.
  • Wrangler deploy sube el código a la red global en ~3 segundos.
  • La latencia es muy baja porque el código corre en el datacenter de Cloudflare más cercano al usuario.

WorkersJS/TS serverless en el edgeAPIs, lógica dinámica, proxies, middleware

5. Buenas prácticas de seguridad aprendidas

Lo que hicimos bien

  • .dev.vars en .gitignore — el token de desarrollo nunca llega a GitHub.
  • Token de producción guardado como secret en Cloudflare (encriptado, write-only).
  • Endpoint de creación (POST /api/shorten) protegido con Authorization: Bearer.
  • Validación de input: URL debe ser http:// o https://, slug solo [A-Za-z0-9_-].
  • Verificación de slug duplicado antes de escribir en KV (evita sobreescribir links existentes).
  • Token generado con openssl rand -hex 24 (96 bits de entropía, no adivinable).

Mejoras posibles para uso real

  • Usar Cloudflare Turnstile en el formulario público para evitar bots.
  • Agregar rate limiting al endpoint de creación (Workers tiene API nativa para esto).
  • Validar que la URL destino no sea maliciosa (consultar Safe Browsing o lista propia).
  • Múltiples tokens con roles (admin vs usuario) en vez de uno único compartido.
  • Audit log de cada creación en un segundo KV o D1 (quién/cuándo).
  • Rotar el AUTH_TOKEN periódicamente (wrangler secret put AUTH_TOKEN).
