Anatomía de un chat con LLM
en el edge de Cloudflare

Qué pasa por debajo cuando le escribís a Llama 3.1: cómo viaja tu mensaje al Worker, cómo el modelo genera la respuesta token a token, y cómo te llegan esos tokens en streaming al navegador en tiempo real.

El caso que vamos a seguir

Le mandás un mensaje al chat. El Worker manda todo el historial a Llama 3.1 (incluido el system prompt), y el modelo te devuelve la respuesta palabra por palabra en formato Server-Sent Events.

Request del navegador cliente → edge 
POST /api/chat HTTP/1.1
Host: chat-ai.dpacheco65.workers.dev
Authorization: Bearer ...token...
Content-Type: application/json

{
  "messages": [
    { "role": "user",
      "content": "Que es Cloudflare Workers?" }
  ]
}
Response del Worker (stream) edge → cliente 
HTTP/1.1 200 OK
Content-Type: text/event-stream
x-model: @cf/meta/llama-3.1-8b-instruct

data: {"response": "Cloud"}

data: {"response": "flare"}

data: {"response": " Workers"}

data: {"response": " es"}

  ... muchos tokens más ...

data: {"usage": { ... }}

data: [DONE]

El flujo completo en un diagrama

Tres zonas: el navegador, el Worker (lógica + auth) y la GPU donde corre Llama. El Worker actúa como un orquestador entre el cliente y el modelo, encadenando el stream sin tocarlo.

El stream visto por dentro

Esto es lo que viaja por el cable: una sucesión de eventos SSE (Server-Sent Events). Cada uno contiene un fragmento de la respuesta. Llegan uno detrás de otro mientras Llama los genera.

Bytes que llegan al navegador (cronológicamente)
data: {"response": "Cloud"}
data: {"response": "flare"}
data: {"response": " Workers"}
data: {"response": " es"}
data: {"response": " una"}
data: {"response": " plata"}
data: {"response": "forma"}
data: {"usage": {"prompt_tokens":85, "completion_tokens":42}}
data: [DONE]

El cliente JavaScript lee cada línea, busca el prefijo data:, parsea el JSON, y concatena response al mensaje del assistant que está renderizando. Cuando ve [DONE], marca el mensaje como completo.

Cada paso, en detalle

Qué hace cada actor del sistema desde que tipeás "Enter" hasta que la última palabra aparece en pantalla.

Usuario · Navegador

Agrega tu mensaje al historial local

El frontend mantiene un array history en localStorage. Al apretar Enter, le hace push al objeto {role: "user", content: ...} y renderiza el mensaje.

Usuario · Navegador

Envía POST con el historial completo

Hace fetch("/api/chat") con method POST, Bearer token, y body JSON {messages: history}. Todo el historial viaja cada turno — el server no recuerda nada entre requests.

Worker

Valida auth y body

Compara Authorization contra env.AUTH_TOKEN. Verifica que cada mensaje tenga un rol válido (system, user, assistant) y contenido entre 1 y 4000 chars.

Worker

Prepend del system prompt

Si el cliente no envía un mensaje system, el Worker antepone uno desde env.SYSTEM_PROMPT. Esto le da personalidad al modelo sin tocar el código.

Worker → AI

Inferencia con streaming

Llama env.AI.run("@cf/meta/llama-3.1-8b-instruct", { messages, stream: true, max_tokens: 1024 }). Devuelve un ReadableStream que va emitiendo chunks.

Workers AI · GPU

Llama genera tokens

El modelo recibe todo el contexto (system + historial). Genera un token, lo envía como evento SSE. Calcula el siguiente, lo envía. Repite hasta llegar a <|eot|> o al límite de tokens.

Worker → Usuario

Passthrough del stream

El Worker devuelve el stream tal cual con header content-type: text/event-stream. No acumula nada — los chunks pasan al cliente a medida que llegan.

Usuario · Navegador

Lee el stream con ReadableStream

Usa response.body.getReader() + TextDecoder. Cada línea con data: se parsea como JSON. Va concatenando response al mensaje del assistant en el DOM y haciendo scroll. Al ver [DONE], marca el mensaje completo y lo guarda en localStorage.

Lo importante de notar

Tres conceptos que diferencian este modelo de un chatbot tradicional con sesión en server.

Stateless puro

El server no guarda conversaciones. El historial vive en el navegador del cliente (localStorage) y se manda completo cada turno. Beneficio: el Worker escala infinito porque no tiene estado.

Stream passthrough

El Worker no acumula tokens. Toma el ReadableStream de la AI y lo retorna al cliente. Esto es lo que hace que las palabras aparezcan a medida que se generan, no todo junto al final.

Modelo en GPU, lógica en CPU

El Worker corre en CPUs distribuidas (todos los datacenters). Llama corre en datacenters con GPUs (menos cantidad, más caros). Cloudflare orquesta los dos transparentemente.

El catálogo de modelos

Workers AI expone decenas de modelos open-source. Se cambia de modelo sin tocar el código: solo se edita MODEL en wrangler.toml y se redeploya.

Sesión en server vs. historial en cliente

Dos enfoques posibles para mantener el contexto de un chat. Este proyecto eligió el segundo.

Sesión en server Stateful · enfoque tradicional

El server guarda la conversación en una DB asociada a un session_id.

POST /chat/abc-123
{ "message": "¿cómo me llamo?" }

// Server:
//   1. Lee historial de DB WHERE session = abc-123
//   2. Le pasa el historial al LLM
//   3. Guarda la respuesta en DB
  • Pro: el cliente solo envía el mensaje nuevo (menos bytes).
  • Pro: se puede analizar / moderar / loggear lo que pasa.
  • Contra: necesitás auth fuerte para identificar usuarios.
  • Contra: ocupás storage por sesión, scaling más complejo.
  • Contra: cuestiones de privacidad (logs de conversaciones).

Historial en cliente Stateless · este proyecto

El cliente manda toda la conversación cada vez. El server es puro pasamanos.

POST /api/chat
{
  "messages": [
    { role:"user", content:"Soy Dario" },
    { role:"assistant", content:"Hola Dario" },
    { role:"user", content:"¿cómo me llamo?" }
  ]
}
  • Pro: cero estado en server, escala infinito.
  • Pro: privacidad mejorada — la conversación no se loggea.
  • Pro: simplísimo de implementar (sin DB).
  • Contra: cada turno envía más bytes (todo el historial).
  • Contra: si el usuario limpia localStorage, perdió todo.
  • Contra: no se puede continuar la conversación desde otro dispositivo.

Una analogía simple

Pensalo como una consultoría telefónica con un experto que tiene amnesia retrograda perfecta: olvida cada llamada anterior. Vos sos el que tiene la memoria.

Cada vez que llamás al experto le tenés que recordar toda la conversación previa: "Mirá, antes te conté que soy Dario, vos me dijiste que me ayudabas con SQL, ahora te quiero preguntar cómo hacer un JOIN". El experto procesa todo el contexto y te contesta en streaming, frase por frase. Cuando cuelgas, él se olvida de todo. Vos guardás la transcripción para la próxima llamada.
