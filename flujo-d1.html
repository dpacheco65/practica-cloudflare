Anatomía de una request CRUD
contra D1 en el edge

Paso a paso de lo que ocurre cuando creás una tarea en la app: del POST del navegador, al prepared statement, a SQLite serverless, y de vuelta con el JSON listo.

El caso que vamos a seguir

Crear una tarea en la app. El navegador manda un POST con el texto, el Worker valida y consulta D1, y devuelve la fila completa con su id, sus timestamps y el flag completed en false.

Request del navegador cliente → edge 
POST /api/tasks HTTP/1.1
Host: tareas-d1.dpacheco65.workers.dev
Authorization: Bearer ...token...
Content-Type: application/json

{
  "text": "Comprar pan"
}
Response del Worker edge → cliente 
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 1,
  "text": "Comprar pan",
  "completed": false,
  "created_at": "2026-05-19 18:04:23",
  "updated_at": "2026-05-19 18:04:23"
}

El flujo completo en un diagrama

Dos zonas: el navegador del usuario y el edge de Cloudflare (donde conviven Worker y D1). Una sola request del usuario genera dos queries SQL internas: un INSERT y un SELECT de vuelta.

Cada paso, en detalle

Lo que el código hace de verdad cuando entra el POST. Mismo Worker, dos queries internas a D1.

Usuario · Navegador

Envía un POST con el texto

El frontend (form de la app) hace fetch("/api/tasks") con method POST, header Authorization: Bearer ... y body JSON {"text": "Comprar pan"}.

Worker

Valida la autenticación

requireAuth() compara el header Authorization contra env.AUTH_TOKEN. Si no coincide, responde 401 sin tocar la DB.

Worker

Parsea y valida el body

Hace request.json(), recorta el text con .trim(), verifica que esté entre 1 y 500 caracteres. Si falla, devuelve 400 con un código de error.

Worker → D1

Prepared statement con bind

Llama env.DB.prepare("INSERT INTO tasks (text) VALUES (?) RETURNING id").bind(text).run(). El ? es el placeholder; el valor real se pasa por bind(), no por concatenación — eso evita SQL injection.

D1 · SQLite

Inserta la fila y aplica defaults

SQLite asigna id = 1 (autoincrement). Aplica los defaults: completed = 0, created_at y updated_at con datetime('now'). La fila ya existe físicamente en la base.

D1 → Worker

Devuelve el id de la fila creada

La cláusula RETURNING id hace que D1 devuelva meta.last_row_id = 1 sin necesidad de una segunda query solo para obtener el id.

Worker → D1

SELECT para devolver la fila completa

Para responder al cliente con created_at y updated_at reales (los puso la DB, no el Worker), se hace un SELECT * FROM tasks WHERE id = ?. Es el "leer lo que acabás de escribir".

Worker → Usuario

Responde 200 con el JSON

Convierte completed de 0/1 a false/true (SQLite no tiene booleans) y devuelve el JSON con los 5 campos. El frontend lo agrega visualmente a la lista.

Lo importante de notar

Tres conceptos clave que separan este modelo de una API tradicional con MySQL/Postgres.

Prepared statements

El ? en el SQL no se reemplaza por concatenación. D1 separa la query del valor, lo que hace imposible SQL injection. Es la única manera correcta de pasar datos del usuario a una query.

D1 también vive en el edge

A diferencia de una DB tradicional en un solo datacenter, D1 está físicamente cerca del Worker. La latencia Worker ↔ D1 es de 2-8 ms, no 50-200 ms como ir a otra región.

Una request, varias queries

El cliente hizo 1 POST. El Worker hizo 2 queries a D1 (INSERT + SELECT) para devolver datos completos. D1 maneja muchas queries por request sin penalidad.

El SQL que ejecuta cada endpoint

Cuatro operaciones CRUD, cuatro patrones SQL. Todos usan bind() para los valores.

GET /api/tasks · listar

SELECT id, text, completed,
       created_at, updated_at
FROM tasks
ORDER BY created_at DESC, id DESC

Lee todas las filas. Usa el índice idx_tasks_created_at para ordenar sin escanear.

POST /api/tasks · crear

INSERT INTO tasks (text)
VALUES (?)
RETURNING id

Inserta solo text. El resto de columnas (id, timestamps, completed) los pone la DB con defaults.

PATCH /api/tasks/:id · editar

UPDATE tasks
SET text = ?,
    completed = ?,
    updated_at = datetime('now')
WHERE id = ?

Construido dinámicamente según qué campos vienen en el body. updated_at se refresca siempre.

DELETE /api/tasks/:id · borrar

DELETE FROM tasks
WHERE id = ?

Si meta.changes === 0, el id no existía y devuelve 404. Si fue 1, responde con {deleted: true, id}.

KV vs D1: cuándo cada uno

El proyecto anterior usó KV (clave-valor); este usa D1 (SQL relacional). Cada uno tiene su lugar.

KV Key-value · El acortador

Diccionario distribuido. Una clave → un valor (string o JSON serializado). Sin queries.

// Crear
env.LINKS.put("slug", JSON.stringify(data))

// Leer
const raw = await env.LINKS.get("slug")

// Borrar
env.LINKS.delete("slug")

// "Listar" requiere recorrer todo
env.LINKS.list({ prefix: "user:" })
  • Ideal para: lookups simples, sesiones, config, caches, feature flags.
  • No sirve para: filtros (WHERE), joins, agregados (SUM, COUNT), orden por campos arbitrarios.
  • Lectura: ~1-5 ms. Escritura: eventualmente consistente (puede tardar segundos en propagar global).

D1 SQL relacional · Las tareas

SQLite serverless. Tablas, columnas tipadas, índices, joins, transacciones, vistas.

-- Listar con filtro y orden
SELECT * FROM tasks
WHERE completed = 0
ORDER BY created_at DESC
LIMIT 20;

-- Agregados
SELECT COUNT(*), AVG(LENGTH(text))
FROM tasks WHERE completed = 1;
  • Ideal para: datos estructurados con relaciones, reportes, búsquedas con criterios.
  • Tradeoff: más complejo, hay que diseñar schema y pensar en índices.
  • Lectura: ~2-8 ms por query. Escritura: fuertemente consistente (lo que escribís lo leés inmediatamente).

Una analogía simple

Pensalo como un bibliotecario que recibe libros nuevos y los ingresa al catálogo.

Llegás a la biblioteca con un libro nuevo para donar. El bibliotecario primero pide tu credencial de socio (auth). Después anota el libro en el catálogo, le asigna un número de inventario (id autoincrement), pone la fecha de ingreso (created_at), y te entrega una ficha con todos los datos del registro recién creado.
