1. Resumen ejecutivo

Completado Workers AI Llama 3.1 8B SSE Streaming Stateless

Qué se construyó

Un chatbot conversacional accesible vía web. El usuario escribe mensajes en una UI minimalista, el Worker los manda a Llama 3.1 con streaming activado, y los tokens generados por el modelo aparecen palabra por palabra en pantalla. El historial de la conversación se guarda en localStorage del navegador (el server no guarda nada).

Métricas y características

Modelo default@cf/meta/llama-3.1-8b-instruct
Tamaño máx por mensaje4000 caracteres
Mensajes máx por turno30 (anti-loop de cliente)
Tokens máx por respuesta1024
Auth de escrituraBearer token (secret en Cloudflare)
Latencia primer token (TTFT)~300-700 ms
Throughput~40-80 tokens/segundo
Persistencia serverCero. Todo en localStorage
CostoFree tier de Workers AI (sin tarjeta)

4. Cómo se llama al modelo

Workers AI expone los modelos a través del binding env.AI. No hay que importar SDKs, no hay que manejar API keys, no hay que crear recursos previos: solo declarar el binding y usarlo. 

const stream = await env.AI.run(
  "@cf/meta/llama-3.1-8b-instruct",
  {
    messages: [
      { role: "system", content: env.SYSTEM_PROMPT },
      { role: "user",   content: "Que es Cloudflare Workers?" }
    ],
    stream: true,
    max_tokens: 1024,
  }
);

return new Response(stream, {
  headers: {
    "content-type": "text/event-stream",
    "cache-control": "no-cache",
  }
});

Parámetros importantes

messagesArray con la conversación. Cada mensaje tiene role (system / user / assistant) y content.
streamtrue = devuelve un ReadableStream en formato SSE. false = devuelve el objeto JSON completo cuando termina.
max_tokensLímite duro de tokens en la respuesta. Llama corta cuando alcanza esto.
temperature(Opcional) 0.0 = determinístico, 1.0 = creativo. Default ~0.7.

5. El streaming SSE en detalle

Cuando stream: true, Workers AI devuelve un stream que sigue el formato Server-Sent Events. Cada evento es una línea que empieza con data: seguida de un JSON, terminada en doble newline.

Lo que viaja por el cable

data: {"response": "Cloud"}

data: {"response": "flare"}

data: {"response": " Workers"}

data: {"response": " es"}

  ... más tokens ...

data: {"usage": {"prompt_tokens": 85, "completion_tokens": 42}}

data: [DONE]

Cómo el frontend lo lee

const r = await fetch("/api/chat", { method: "POST", ... });
const reader = r.body.getReader();
const decoder = new TextDecoder();
let buffer = "";

while (true) {
  const { value, done } = await reader.read();
  if (done) break;
  buffer += decoder.decode(value, { stream: true });
  const lines = buffer.split("\n");
  buffer = lines.pop();
  for (const line of lines) {
    if (!line.startsWith("data:")) continue;
    const payload = line.slice(5).trim();
    if (payload === "[DONE]") break;
    const json = JSON.parse(payload);
    if (json.response) {
      // Concatenar al mensaje del assistant en el DOM
      body.textContent += json.response;
    }
  }
}

El buffer se mantiene entre lecturas porque un chunk de red puede cortar un evento por la mitad. Sólo procesamos líneas completas (terminadas en \n).

7. Frontend (cómo funciona el HTML)

El HTML está incrustado en el Worker (en homePage()). No hay archivos separados: todo se sirve desde el endpoint GET /. Esto simplifica el deploy porque un solo Worker tiene frontend + API.

Estado en el cliente

VariableDónde vivePara qué
historylocalStorage chat-historyArray de mensajes. Se serializa como JSON.
TokenlocalStorage token-chatEl Bearer token, persistido para no escribirlo cada vez.
busyVariable en memoriaFlag para evitar enviar mientras hay un request en curso.

Flujo de la UI

  1. Al cargar la página, lee localStorage y renderiza los mensajes previos.
  2. Al apretar Enter, agrega el mensaje del usuario al array y lo renderiza inmediatamente.
  3. Hace POST y muestra un placeholder "pensando" con animación.
  4. Cuando llegan los primeros bytes, reemplaza el placeholder por un mensaje vacío y va apendeando texto.
  5. Al [DONE], marca el mensaje como completo, lo guarda en localStorage.
  6. Habilita de nuevo el input y devuelve foco al textarea.

8. Catálogo de modelos disponibles

Workers AI expone decenas de modelos open-source. Se cambia de modelo sin tocar el código: solo editar MODEL en wrangler.toml y redeployar.

En uso
Llama 3.1 8B Instruct
@cf/meta/llama-3.1-8b-instruct

Buen balance velocidad / calidad. Default del proyecto. Para chat conversacional general.

Más capaz
Llama 3.3 70B Instruct (fp8 fast)
@cf/meta/llama-3.3-70b-instruct-fp8-fast

Modelo 70B cuantizado. Mejor razonamiento, código y consultas complejas. Consume más Neurons.

Más rápido
Llama 3.2 3B Instruct
@cf/meta/llama-3.2-3b-instruct

Modelo chico. Respuestas más veloces y menos costo. Bueno para tareas simples / clasificación.

Visión
Llama 3.2 11B Vision Instruct
@cf/meta/llama-3.2-11b-vision-instruct

Acepta imágenes además de texto. Le pasás una foto en base64 y te describe / responde sobre ella.

Embeddings
BGE Base EN
@cf/baai/bge-base-en-v1.5

Convierte texto en vectores (768 dimensiones). Base para búsqueda semántica y RAG con Vectorize.

Text-to-image
Stable Diffusion XL Lightning
@cf/bytedance/stable-diffusion-xl-lightning

Generación de imágenes. Prompt en inglés, devuelve PNG. Muy rápido (4-8 pasos).

Cómo cambiar de modelo sin redeployar código

  1. Editar wrangler.toml:
[vars]
MODEL = "@cf/meta/llama-3.3-70b-instruct-fp8-fast"   # nuevo modelo
  1. Ejecutar npx wrangler deploy (solo actualiza las vars, no recompila).
  2. El próximo request al chat ya usa el modelo nuevo.

9. Cómo usarlo

Desde el navegador (más simple)

  1. Abrir chat-ai.dpacheco65.workers.dev
  2. Pegar el AUTH_TOKEN en el campo arriba a la derecha (queda guardado en localStorage)
  3. Escribir el mensaje y apretar Enter (o Shift+Enter para newline)
  4. Ver la respuesta aparecer palabra por palabra
  5. "Limpiar" arriba para empezar una conversación nueva

Desde la línea de comandos (PowerShell)

$token = "TU_AUTH_TOKEN"
$body = @{
  messages = @(
    @{ role = "user"; content = "Holaa, decime algo en una palabra" }
  )
} | ConvertTo-Json -Depth 4

Invoke-RestMethod -Method Post `
  -Uri "https://chat-ai.dpacheco65.workers.dev/api/chat" `
  -Headers @{ Authorization = "Bearer $token"; "Content-Type" = "application/json" } `
  -Body $body

Desde curl (con streaming visible)

curl -N -X POST https://chat-ai.dpacheco65.workers.dev/api/chat \
  -H "Authorization: Bearer TU_AUTH_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"messages":[{"role":"user","content":"Hola"}]}'

El flag -N desactiva el buffer de curl para ver los tokens llegar en vivo.

10. Aprendizajes clave

  • Workers AI binding: se declara con [ai] en wrangler.toml, sin necesidad de crear recursos previos (a diferencia de KV/D1/R2).
  • Inference con LLM: env.AI.run(model, opts) — interfaz tipo OpenAI Chat Completions, con messages, stream, max_tokens, etc.
  • Mensajes con roles: system (instrucción al modelo), user (mensajes de la persona), assistant (respuestas previas del modelo). El historial es cómo se mantiene contexto.
  • System prompt como var: ponerlo en [vars] permite cambiar la personalidad del bot sin tocar código (solo wrangler deploy).
  • Streaming SSE: el server emite data: {json}\n\n. El cliente usa ReadableStream.getReader() + TextDecoder + buffer entre chunks.
  • Stream passthrough: el Worker no acumula los tokens. Toma el stream que devuelve env.AI.run() y lo retorna tal cual en la Response.
  • Neurons: unidad de costo de Workers AI. Plan free incluye una quota diaria suficiente para uso de prueba.
  • Stateless puro: cero estado en server, todo el historial en localStorage del cliente. Beneficios: escala infinito, mejor privacidad. Costo: cada turno envía más bytes.
  • Catálogo de modelos: Llama 3.1 8B/70B/3B, Vision, BGE embeddings, Stable Diffusion. Se cambia con un edit de [vars].

11. Seguridad

Lo que se hizo bien

  • .dev.vars en .gitignore — el token de desarrollo nunca llega a GitHub.
  • Token de producción como secret en Cloudflare (encriptado, write-only).
  • Endpoint POST /api/chat protegido con Authorization: Bearer.
  • Validación de cada mensaje: roles permitidos, contenido entre 1 y 4000 caracteres.
  • Límite MAX_MESSAGES = 30: evita bucles que envían historiales infinitos.
  • Token generado con openssl rand -hex 24 (96 bits de entropía).
  • Códigos HTTP correctos (401, 400) que no filtran información extra.
  • max_tokens: 1024 server-side limita la respuesta aunque el cliente pida más.

Mejoras posibles para uso público

  • Agregar Cloudflare Turnstile al formulario para evitar bots.
  • Implementar rate limiting por IP (Workers tiene API nativa).
  • Moderar contenido con Llama Guard u otro modelo antes y/o después del modelo principal.
  • Loggear (sin contenido sensible) los abusos detectados para análisis posterior.
  • Validar que el system prompt enviado por el cliente (si se permite) no contenga jailbreaks conocidos.
  • Múltiples tokens con quota distinta por usuario.
  • Rotar el AUTH_TOKEN periódicamente.

12. Limitaciones del modelo

  • Cutoff de conocimiento: Llama 3.1 fue entrenado con datos hasta mediados de 2024. No conoce noticias / eventos posteriores.
  • Alucinaciones: como cualquier LLM, puede inventar datos con confianza. No usarlo para hechos verificables sin doble check.
  • Idiomas: muy bueno en inglés, bueno en español. Otros idiomas pueden tener calidad variable.
  • Sin acceso a internet: no busca cosas en tiempo real. Si le preguntás "qué hora es" o "cuál es la última noticia", inventa.
  • Sin tools / function calling: este Worker no expone tools. Llama responde solo con texto, no puede llamar APIs externas.
  • Contexto limitado: el modelo tiene una ventana de contexto (8K tokens para 3.1 8B). Conversaciones muy largas eventualmente pierden el inicio.
  • No multimodal en este modelo: para subir imágenes hay que cambiar a llama-3.2-11b-vision-instruct.

13. Mejoras posibles

  1. Function calling / tools — Permitir que Llama decida llamar a otras APIs (clima, búsqueda, calculadora). Se hace pasando definiciones de tools en el request.
  2. RAG (Retrieval-Augmented Generation) — Indexar tus documentos con embeddings (BGE) en Vectorize y meterlos como contexto. El chat responde con info tuya.
  3. Multimodal (imágenes) — Cambiar a Llama Vision. Endpoint que acepta {messages, image} y el modelo responde sobre la imagen.
  4. Persistencia opcional en D1 — Endpoint POST /api/conversations que guarda el historial actual con un id y permite recuperarlo desde otro dispositivo.
  5. Múltiples conversaciones — UI con tabs o sidebar para mantener varios chats abiertos en paralelo.
  6. Exportar conversaciones — Botón para bajar el historial como Markdown o JSON.
  7. Edit / regenerate — Permitir editar el último mensaje del usuario y regenerar la respuesta. O regenerar la respuesta sin editar el prompt (para ver variaciones).
  8. Selector de modelo en la UI — Permitir al usuario elegir entre Llama 8B (rápido), 70B (capaz) y 3B (más rápido aún) desde el frontend.
